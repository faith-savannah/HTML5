Chapter One: A brief history of Markup

  • HyperText Markup Language(HTML) was the brainchild of Sir Tim Berners-Lee

  • The first official specification was HTML 2.0, published by the IETF.Thus there was never HTML 1.0

  • The role of the IETF was superceded by the W3C, the World Wide Web Consortium, where subsequent iterations of the HTML standard have been published at http://www.w3.org.

  • After subsequent revisions from HTML 2.0 to HTML 4.01, the next revision bore XHTML 1.0

  • eXtreem HTML 1.0/XHTML 1.0 was different from HTML in the syntax only whereby XHTML required a stricter adherance to the rules of XML markup language.

  • WHATWG stands for WebHypertext Application Technology Working Group.It propagated the idea of extending HTML to allow the creation of web applications, opposed to the path the W3C took of XHTML 2.0 which was backward incorpartible.

  • HTML5 is a specification that merged two specifications: Web Forms 2.0 and Web Apps 1.0. Both specifications were from the WHATWG and were intended to extend HTML

  • There are two groups working on HTML5. The WHATWG is creating an HTML5 specification using its process of “commit then review. ” The W3C HTML Working Group is taking that specification and putting it through its process of “review then commit.”

  • XHTML is dead.

  • HTML5 follows a design principle of “Priority of constituencies,” which states, “In case of conflict, consider users over authors over implementers over specifiers over theoretical purity.”

  • The minimum information required to ensure that a browser renders using standards mode is the "HTML5 doctype".In fact, that’s the only reason to include the doctype at all

Chapter Two: The design of HTML5

  • One of the key principles of HTML5(and the reason why it gained favour over XHTML) is to “Support existing content.”

  • Some of the other design principles include “Do not reinvent the wheel,” and “Pave the cowpaths.”

  • The principle “Pave the cowpaths”, put in another way is:“If it ain’t broke, don’t fix it.” This helps in achieving the "Priority of constituencies" design principle..

  • The HTML5 specification needs to be powerful enough to support the creation of web applications while still needs to support existing content, even if most existing content is a complete mess.Thus need to strike a balance between this concerns.

  • The HTML5 specification doesn’t just declare what browsers should do when they are processing well-formed markup, but also defines what browsers should do when they are dealing with badly formed documents.The latter is important for browser makers

  • HTML5 takes the approach of keeping it simple and straight forward in its doctype declaration and meta tags.This is illustrated in creating this document within the "head" tag

  • In HTML5 the white space at the start of a line isn’t significant. Nor is the application of case sensitive tags.having to enforce preferred writing style is a personal choice unlike in some languages such as python.

  • You also dont have to include the "type='text/css' " attribute of a link tag nor the same attribute in a "script" element in HTML5

  • No deprecated elements but obsolete elements for purposes of backward compartibility. If used though(the obsolete elem i.e) it leads to issue of non-conformance of the html doc

  • Some obsolete elements include the frame, frameset, acronym(replaced by “abbr”) and noframes elements

Chapter Three: Rich media

  • Html5 introduces rich media elements native to the browsers to replace the plugins.Plugins fill the holes on the web. They make it relatively easy to get games, films, and music online.

  • Plugin technologies are not open. They are not created by the community. They are under the control of individual companies.And this can be restrictive to developers thus the advantage of rich media elements introduced.
